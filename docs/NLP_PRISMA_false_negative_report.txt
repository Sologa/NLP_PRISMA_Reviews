一句話結論

目前 pipeline 的主要問題不是單一 reviewer，而是 serialization + generic Stage 1 prompt +
recall‑hostile arbitration 的交互作用。尤其在 2307.05527 中，topic_definition 被混進 inclusion
text 後，scope 被錯收窄成「必須顯式談 ethics」    ；在 2409.13738 與 2601.19926 中，證據不足被過早
視為排除證據；在 2511.13936 中，operational definition 本身也確實過窄。


最優先三項修法

第一，修改 topic_pipeline.py 的 Stage 1 aggregation policy，將 disagreement 與 uncertainty 預
設轉為 maybe。第二，重做 criteria serialization，停止把 topic_definition prepend 到 inclusion
string。第三，用 Stage 1 專用 prompt 取代目前 generic vendor prompt，明寫 lack of evidence !
= evidence of exclusion。


Executive summary

這份報告是把我先前的中文分析與後續實作計畫整理成一份可直接執行的設計文件。核心結論很
明確：目前 screening workflow 的主要失血點不是單一 reviewer，而是 Stage 1 把不適合在 title
+ abstract 階 段 硬 判的條 件， 以 過 度 扁 平的 criteria serialization 與 過 度 generic 的 runtime
prompt 交給一個 recall‑hostile 的 aggregation policy 來做最終 gate。依 handoff 與 repo 內本
地分析，四篇 review 合計 final FN = 153，其中 117 個在 Stage 1 就流失，約占 76%。這意味著，若
Stage 1 不被重設為真正的 high‑recall gate，後面任何 full‑text reviewer 改進都只能補救少部分損
失。

3. 目前 runtime pipeline 到底如何放大 false negatives

這一節直接對照 runtime code，而不是只討論高層概念。因為目前 false negatives 的多數根因，不
是抽象的「模型太保守」，而是已經具體寫在程式裡：criteria 被怎麼序列化、senior 何時介入、final
verdict 怎麼導出、missing full text 怎麼記錄。只要把這些程式邏輯攤開，很多本地分析中的觀察都
會變得更可檢驗。


3.1 Criteria serialization：為什麼 2307 會被收窄成「paper 必須明談 ethics」

在 topic_pipeline.py 的 _criteria_payload_to_strings 中， 程 式 先 讀 取 topic_definition， 再 把
它直接 prepend 到 reviewer 看到的 inclusion text。這個動作本身就把背景 framing 與 formal
eligibility criteria 混 成 同 一 層。 對 一 般 topic 也 許 只 是 語 義 上 不 乾 淨； 對 2307.05527 這 種
topic_definition 本身高度 ethics‑heavy 的 review，則會直接把 scope 壓窄成 reviewer 眼中必須滿
足的條件。

2511 是 四 篇 裡 最 接 近 genuine criteria problem 的 一 篇。JSON 與 source markdown 都 把

agreement derived rankings 等情境。於是，reviewer 很容易採用一個過窄、但又完全符合字面理

negative，只有 2 個是 arbitration 壓掉。這表示 2511 不是主要的 senior‑override 問題，也不是某
一位 junior 特別差，而是兩位 junior 都沿著同一個過窄邊界在判。這與 2307 或 2601 的結構完全不
同。

公 開 metadata 對 這 個 判 斷 很 有 幫 助。lotfian2016retrieving 的 abstract 直 接 寫 出 preference
learning、pairwise comparisons 與 affective recognition；parthasarathy2016using 寫 的 是
rank‑based emotion classifiers；han2020ordinal 明 講 ordinal speech emotion recognition
in customer service calls；parthasarathy2018preference 又 是 preference learning with
qualitative agreement。 若 gold 把 這 些 視 為 in‑scope， 則 現 有 criteria 的 machine‑facing
operational boundary 顯然沒有把它們明確納入。

因 此，2511 的 首 修 不 是 arbitration hotfix， 而 是 criteria surgery。 應 明 示 下 列 幾 件 事： 第
一，ordinal formulations 可 以 算 preference learning； 第 二，rank‑based learning 可 以 算
preference learning；第三，relative labels 與 qualitative agreement derived rankings 可以算
preference signal；第四，speech emotion、customer service calls、canonical speech datasets
可以視為 audio‑domain evidence；第五，若 audio 是 core modality，multi‑modal work 仍可納
入。只要這些邊界被明寫，reviewer 的 shared under‑inclusion 就會大幅下降。

5.4 2601.19926：為什麼這篇是 sparse metadata、over‑conservative Stage 1 與 missing full
text 的混合型問題

2601 是 四 篇 裡 最 混 合 的 一 篇。 從 criteria markdown 看，review 自 身 已 明 確 寫 出 relevance
assessment primarily via title, then abstract/ full text when necessary， 並 把 cutoff date 與
publication type 等 metadata 條件分開。可是 runtime pipeline 在 Stage 1 並沒有利用這個「when
necessary」 的 設 計 精 神， 而 是 仍 把 扁 平 化 criteria 丟 給 title+abstract reviewer。 當 metadata
本身又常是 citation‑like 或極短 abstract 時，reviewer 若沒在表面看到 Transformer、syntax、
probing 等強詞，就很容易保守排除。

mccoy_right_2019 就是最典型的 sparse metadata 例子：metadata 幾乎只有 bibliographic line，
沒有足夠 abstract 讓 reviewer 判定 syntactic heuristics / NLI 與 Transformer/syntax probing 的
關係。相反地，Tenney:etal:2019b 與 Warstadt:etal:2020 的 abstract 已明確顯示 contextualized
representation、BERT、sentence structure、grammatical phenomena 等訊號；它們若在後續
被 標 記 成 exclude (missing_fulltext)， 那 就 是 retrieval/ path resolution 問 題， 而 不 是 Stage 1
semantic misunderstanding。

本地分析對 2601 的本質描述相當準：它是一半 reviewer 在 sparse metadata 上過度保守，一半是
arbitration / aggregation 放大了保守傾向，外加少量 full‑text retrieval failure。我的補充是：2601
其實是最能凸顯「semantic failure 與 operational failure 必須分開記錄」的一篇。因為只要仍把
missing_fulltext 直接記成 exclude，就會混淆你到底該優先去修 prompt policy，還是該優先去修
full‑text path resolution。

6. 我不同意或要補強本地分析的地方

整體上，我大致同意 repo 內本地分析的方向。不過若要把它變成一份真正可實作的 engineering
diagnosis，還有幾個地方需要講得更精準。以下列出我最明確的三項修正。

第一，我不同意把 2307 首要歸因到「criteria 寫不好」。更合理的說法是：原始 source criteria
並 沒 有 要 求 explicit ethics discussion； 真 正 把 scope 收 窄 成 「paper 必 須 談 ethics」 的 是
machine‑facing serialization。這個差別很重要，因為它決定你應該先改 criteria_jsons 還是先改
topic_pipeline.py。我的答案是：先改 pipeline。

第 二， 我 不 建 議 把 JuniorMini anchor 當 成 長 期 設 計。 它 可 以 作 為 短 期 hotfix， 因 為 Stage 1
aggregate recall 的確較好；但它無法根治 2511 的 criteria‑boundary 問題，也無法修掉 2307 的
serialization bug。長期上應修 decision rule、prompt 與 criteria projection，而不是把 recall 依
賴在單一 reviewer personality 上。

第三，我會把「同一套 criteria 被直接套到 Stage 1 與 Stage 2」升級成第一級 architecture 缺陷。
本地分析已經指出這點，但我認為它應該被明確命名為 criteria observability mismatch。因為這不
只是 reviewer 行為保守，而是系統根本沒有區分哪些條件在 title+abstract 可觀察、哪些條件只能在
full text 或 metadata 層穩定判定。

7.3 用 Stage 1 專用 prompt 取代目前 generic vendor prompt

建 議 直 接 重 寫 title_abstract_reviewer.py 的 generic_prompt_all / generic_prompt_any， 把

請嚴格遵守下列規則：
1. 你只能依據 title 與 abstract 判斷；不得假設全文內容。
2. topic_definition 只是背景說明，不是硬性 inclusion criterion。
3. 若某條 criterion 在 title/abstract 中不可觀察，請標記為 UNCLEAR，不得因此直接判定為 NO。
4. lack of evidence for inclusion != evidence for exclusion。

7.4 重寫 Stage 2 prompt，但把目標定義為「可稽核」而不是再次保守

條 criterion 出問題的狀況。這部分主要影響 fulltext_reviewer.py、basic_reviewer.py 與可能的
downstream schema handling。
你現在執行的是 Stage 2（full‑text final review）。

請逐條輸出 criteria 狀態，而不是只給單一整體分數：

1. 對每條 inclusion / exclusion 回傳 YES / NO / UNCLEAR。
